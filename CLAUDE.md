# CLAUDE.md

`rice-knowledge-map` で作業する Claude Code 向けの唯一の作業指示です。

## 1. プロジェクトの目的

実画像マップ型の稲作ナレッジ記録PWAを、自分の田んぼと家族共有で実用できる状態まで仕上げる。

- 主役は実画像マップ。普通の農業日誌や管理ダッシュボードに寄せない。
- 現場で「写真を撮る → 話す → 保存する → 地図に残る」を短くする。
- 入水口・出水口・水路・異常箇所・過去の対応を、場所と結びつけて残す。
- 初期MVPでは、売上管理、帳票、AI画像診断、センサー連携、全国向けSaaS化を優先しない。

## 2. 読む順番と一次情報

作業前に、必要な範囲だけ次の順で確認する。

1. 現在のコード
2. ユーザーがこの会話で明示した実機確認結果・判断
3. `tasks/TASKS.md`（進捗、実機確認、残タスク）
4. `docs/REQUIREMENTS.md`（MVP要件）
5. `docs/DATA_MODEL.md`（データ構造・状態値）
6. `docs/UIUX_IMPLEMENTATION_ROADMAP.md`（次のUI/UX改善順）
7. `README.md`（起動方法・全体概要）
8. `supabase/README.md`（migration適用状況）

古いPR本文、過去コミット、会話ログ、削除済み文書は補助情報であり、現在の仕様ではない。コードとユーザー判断が食い違う場合は、ユーザー判断を `TASKS.md` に反映してから作業する。

## 3. 文書の運用

- 新しいMarkdownを原則作成しない。
- 監査表、画面棚卸し、実装案、Hook案、レビュー用チェックリストはPR本文またはIssueに置く。恒久docsにしない。
- 同じ事実を複数文書へ複製しない。
- 実機確認結果は `tasks/TASKS.md` だけに記録する。
- 次に何を改善するかは `docs/UIUX_IMPLEMENTATION_ROADMAP.md` だけに記録する。
- 現在のデータ構造・状態値は `docs/DATA_MODEL.md` を正とする。
- 新しい恒久文書が必要だと判断した場合は、作成前に目的・既存文書では不足する理由・更新責任者をPR本文へ書く。

## 4. 作業の進め方

- 1 PR = 1目的。UI、DB、依存関係、大規模構造変更を混ぜない。
- 非自明なUI/UX変更や構造変更は、実装前にPlan Modeで対象画面、変更ファイル、壊してはいけない挙動、確認方法を整理する。
- Plan Modeの成果物は会話またはPR本文に置き、MDを増やさない。
- UI変更は、現行画面を実際に読み、Previewまたはスクリーンショットで確認する。
- モバイル操作を変える場合は、必要なiPhone実機確認項目を `TASKS.md` に追記または更新する。
- 実装後は `npx tsc --noEmit`、`npm run lint`、`npm run build`、変更ファイル再読を行う。
- UI作業とDB作業を混ぜない。

## 5. UI判断

- 実画像マップを常に主役にする。
- 文字、ラベル、フィルター、カードを増やして問題を隠さない。
- 写真・音声・場所・対応状況が短い操作でつながる導線を優先する。
- 片手操作、画面端の安全領域、iPhone Safariのviewport変化を前提にする。
- 画面の「最初に押す操作」は一つに絞る。

## 6. MapCanvasで壊してはいけないこと

- 田んぼ登録は「場所合わせ → 輪郭描画 → 名前入力 → 保存」の順を保つ。
- 新規田んぼ保存後は通常閲覧へ戻る。描き直し成功後は対象田んぼを選択して表示する。
- 描画中の1本指操作はブラウザのページ操作へ漏らさず、2本指の地図ズームは維持する。
- 田んぼ一覧のプレビューと正式選択を混同しない。
- `returnToBrowse()` は途中の描画・プレビュー・保留状態を残さない単一の退避口として扱う。
- MapLibre Markerの楽観追加、DB IDへの差し替え、失敗時ロールバックを壊さない。
- 現在のコードが詳細仕様の一次情報であり、ここへ中間実装の説明を増やさない。

## 7. 承認が必要な操作

ユーザー承認なしに実行しない。

- 既存ファイル削除
- mainへの直接push、PR merge、force push、branch削除
- DB migration適用、既存データやRLSの変更、Storage公開設定変更
- Vercel本番設定、環境変数、ドメイン、課金に関わる変更
- 大規模リファクタリングまたは目的外の機能追加

## 8. 現在の優先順位

1. 本番・iPhone実機で分かった導線課題を、最小単位のUI/UX PRで直す。
2. 初回利用、マップ通常閲覧、田んぼ詳細から記録、記録の開始・保存までの導線を整える。
3. 構造整理は、UI/UX改善を妨げる具体的な問題が出たときだけ、そのPRの最小範囲で行う。
4. T-048、T-051以降の任意機能は、主要導線を改善した後に判断する。
