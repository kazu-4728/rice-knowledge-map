# rice-knowledge-map

実画像マップ型 稲作ナレッジ記録アプリ。

このリポジトリは、田んぼの暗黙知を実画像マップ上に固定し、写真・音声・位置情報・家族コメント・対応状況として残すアプリを作るためのものです。

目的はテストモックではありません。最終的に、自分の田んぼで実際に使えるPWAアプリとして完成させます。UI確認やモック作成は、そのための途中工程です。

---

## 1. 最重要コンセプト

このアプリは、普通の農業日誌アプリではありません。

中心は、作業一覧や日報ではなく、実画像マップです。

残すべき情報は、以下のような「場所に結びついた現場知識」です。

- 田んぼ区画
- 入水口
- 出水口
- 水路
- 詰まりやすい場所
- 水が抜けにくい場所
- 雑草が出やすい場所
- 畦の崩れ
- 異常箇所
- 過去の対応履歴
- 家族・作業者のコメント

判断に迷ったら、次を優先します。

1. 現場で使いやすいか
2. 実画像マップ上に場所の知識が残るか
3. 家族や次世代に伝わるか
4. 後から見返して意味があるか
5. データ構造が将来拡張に耐えるか
6. 実装が過剰になっていないか

---

## 2. UIの最優先ルール

参照モック画像を、最上位のデザイン基準とします。

既存ドキュメントの要素リストよりも、参照モック画像の見た目、余白、質感、画面密度、スマホアプリ感を優先してください。

すべての要素を画面に詰め込む必要はありません。参照画像の品質を下げる要素は、省略・縮小・下部シート化・別画面化して構いません。

禁止するUI:

- 古い業務システム風UI
- 管理ダッシュボード風UI
- 表や一覧を主役にした画面
- カードをただ並べるだけの画面
- 大きすぎるヘッダー
- 多すぎるフィルター
- 多すぎるラベル常時表示
- グレー背景に白カードを並べるだけのHTMLモック風UI
- 天気カード、予定表、同期ステータス、データ出力などが主役に見えるUI

優先するUI:

- フルスクリーンに近い実画像マップ
- 薄く軽いヘッダー
- 地図上のピンと田んぼ区画
- 下部ボトムシート
- フローティング操作ボタン
- 写真で記録 / 音声メモへの太い導線
- 片手操作しやすいタップ領域
- 余白、角丸、影、半透明、重なりのある現代的なスマホアプリUI

## 3. 技術方針

標準構成は以下です。

```text
Frontend : Next.js App Router
Language : TypeScript
UI       : React + Tailwind CSS
Map      : MapLibre GL JS
Base map : 国土地理院 空中写真タイル
Backend  : Supabase Auth / Postgres / Storage / RLS
Deploy   : Vercel Preview → Production
```

方針:

- 最終実装はNext.jsで行う。
- UI確認もNext.js内で行う。
- HTMLだけの別モックは原則作らない。
- Tailwind CSSを標準にする。
- Vanilla CSSの肥大化は禁止。
- `globals.css` はリセット、Tailwind、MapLibreなど必要最小限にする。
- UIは共通コンポーネントとTailwindクラスで統一する。
- 地図APIに業務データを依存させない。
- 田んぼ、固定ポイント、記録、写真、音声、コメントは自前DBに保存する。

## 4. 初期MVPで作るもの

初期MVPでは、全国向けSaaSや高度な分析機能を狙いません。

まず、自分の田んぼで実際に使い、家族間で場所と状態を共有できる最小実用版を作ります。

初期MVPの中心機能:

- 実画像マップ表示
- 田んぼ区画または中心点の登録
- 入水口、出水口、水路、注意箇所の登録
- 写真撮影または写真アップロード
- 写真の位置情報取得
- スマホGPSによる現在地取得
- ピン位置の手動補正
- 音声メモ
- AI文字起こし
- AIによる記録内容整理
- 保存前確認画面
- 地図上の記録ピン表示
- 記録詳細画面
- 家族コメント
- 対応済み / 未対応管理

後回しにするもの:

- 作業分析
- 売上管理
- 帳票出力
- 補助金関連
- AI画像診断
- センサー連携
- 完全ハンズフリー操作
- 全国向けSaaS化

## 5. 実装順序

実装は、見た目と体験を固めてから、保存処理と本番機能へ進みます。

1. 参照モック画像を基準に、メインマップ画面のUI品質を固める
2. Next.js + Tailwind CSSでスマホアプリらしいApp Shellを作る
3. 実画像マップ画面を作る
4. 田んぼ区画、入水口、出水口、注意箇所、記録ピンを表示する
5. 下部ボトムシートと記録導線を作る
6. 写真で記録する画面を作る
7. 音声メモ画面を作る
8. 保存前確認画面を作る
9. 記録詳細画面を作る
10. Supabaseスキーマを確認する
11. Supabase接続と保存処理を実装する
12. Vercel Previewでスマホ実機確認する
13. 本運用へ進む

UIが参照モック画像から離れている状態で、DB接続や機能追加へ進まないでください。

## 6. エージェント作業ルール

作業前に必ず確認すること:

- 今の作業目的は何か
- 対象画面はどこか
- UI作業か、機能作業か、DB作業か
- 参照モック画像から離れていないか
- 余計な機能を追加していないか

UI作業時のルール:

- 参照モック画像を最優先する。
- 既存MDの要素リストを機械的に全部詰め込まない。
- まず1画面を基準品質まで上げる。
- スクリーンショットまたはPreview URLなしで完了扱いにしない。
- 完了報告では、変更ファイル、確認結果、まだ違う点を明記する。

禁止:

- 目的外の画面追加
- 目的外の機能追加
- UI作業とDB作業の混在
- 承認なしの重要設定変更
- 承認なしの公開設定変更

ただし、ユーザーが明示的に許可した場合は、その指示を優先します。

---

## 7. 現在の最優先課題

現在の最優先課題は、メインマップ画面を参照モック画像に近いスマホアプリUIへ引き上げることです。

見るべき点:

- 実画像マップが主役に見えるか
- 古い管理画面に見えていないか
- ヘッダーが重すぎないか
- 文字やラベルが多すぎないか
- 下部ボトムシートが自然か
- 写真・音声で記録する導線が強いか
- 片手操作できそうか
- 画面全体に現代的なアプリ感があるか

この課題が合格するまで、機能追加で見た目の弱さをごまかさないでください。
